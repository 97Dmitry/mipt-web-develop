# Техническое задание

## 1. Общая информация

- **Проект:** интернет-магазин лампочек для завода-производителя
- **Формат реализации:** веб-приложение на основе микросервисной архитектуры
- **Предметная область:** продажа 20 SKU лампочек через витрину магазина и управление ассортиментом и заказами через админ-панель
- **Целевая аудитория:** конечные покупатели лампочек и сотрудники завода, управляющие каталогом и заказами
- **Основная цель:** предоставить покупателю понятный путь от просмотра каталога до оформления заказа, а администратору — инструменты для управления товарами, остатками, категориями и статусами заказов

## 2. Границы решения

### 2.1 Что входит в MVP

- каталог товаров с фильтрами, поиском и сортировкой
- карточка товара с полным описанием и характеристиками
- корзина без обязательной регистрации пользователя
- оформление заказа в один шаг
- экран подтверждения заказа
- отдельная админ-панель с авторизацией по JWT
- управление товарами, категориями, остатками и статусами заказов
- документация по трем микросервисам
- прототипы всех ключевых экранов в Figma
- Postman-коллекции с примерами API

### 2.2 Что не входит в MVP

- личный кабинет покупателя
- онлайн-оплата
- промокоды, бонусы, избранное, отзывы, сравнение товаров
- push/email/SMS-уведомления
- асинхронные очереди, брокер сообщений, event sourcing
- отдельный auth-service, payment-service, notification-service

## 3. Архитектура и технологические ограничения

### 3.1 Микросервисы

Система состоит из трех REST-микросервисов с изолированными базами данных:

1. `product-service`
   Отвечает за каталог, категории, карточки товаров, изображения и остатки.
2. `order-service`
   Отвечает за корзину, оформление заказов, позиции заказа и историю статусов.
3. `admin-panel-service`
   Отвечает за авторизацию администратора и BFF-поверхность админки, агрегируя данные из `product-service` и `order-service`.

### 3.2 Интеграции

- все межсервисные вызовы выполняются синхронно по HTTP/REST
- `admin-panel-service` не хранит каталог и заказы у себя, а проксирует и агрегирует операции из доменных сервисов
- `order-service` при создании заказа запрашивает актуальные данные о товаре и остатке у `product-service`
- после успешного оформления заказа `order-service` уменьшает остаток в `product-service` через защищенный сервисный endpoint

### 3.3 Базовые технические решения

- СУБД для каждого сервиса: PostgreSQL 16
- формат обмена: JSON
- время в API и БД хранится в UTC
- денежные значения хранятся в копейках в `integer`
- аутентификация администратора: JWT access token
- авторизация в рамках MVP: одна роль `admin`

## 4. Пользовательские роли

### 4.1 Гость

Гость может:

- просматривать каталог
- фильтровать, искать и сортировать товары
- открывать карточку товара
- добавлять товар в корзину
- изменять количество товара в корзине
- оформлять заказ без регистрации
- видеть страницу подтверждения заказа

### 4.2 Администратор

Администратор может:

- войти в админ-панель
- просматривать список товаров
- создавать, редактировать и деактивировать товары
- обновлять остатки
- управлять категориями
- просматривать список заказов
- открывать карточку заказа
- изменять статус заказа

## 5. Функциональные требования к витрине магазина

### 5.1 Экран каталога

Обязательные элементы:

- шапка с логотипом, ссылкой на каталог и кнопкой корзины
- поисковая строка
- фильтры по категории, типу цоколя, мощности, цветовой температуре, наличию
- сортировка по названию и цене
- сетка карточек товаров
- бейдж остатка или отсутствия товара
- пагинация или кнопка "Показать еще"

Поведение:

- список фильтруется без перезагрузки страницы
- при переходе в карточку товара сохраняются активные фильтры каталога
- если товары не найдены, показывается empty state

### 5.2 Экран карточки товара

Обязательные элементы:

- название товара
- артикул
- цена
- изображения товара
- краткое и полное описание
- характеристики: категория, цоколь, мощность, цветовая температура, световой поток
- текущее наличие
- выбор количества
- кнопка добавления в корзину
- блок "Назад в каталог"

Поведение:

- если остаток равен нулю, кнопка добавления в корзину отключена
- количество в форме не может превышать остаток
- после добавления товара пользователь получает уведомление и может перейти в корзину

### 5.3 Экран корзины

Обязательные элементы:

- таблица или список позиций
- название, цена, количество, сумма по позиции
- кнопки увеличения/уменьшения количества
- удаление позиции
- итоговая сумма
- кнопка перехода к оформлению заказа
- empty state для пустой корзины

Поведение:

- изменение количества мгновенно пересчитывает сумму
- количество не может быть меньше 1
- если корзина пустая, оформление заказа недоступно

### 5.4 Экран оформления заказа

Обязательные поля формы:

- `customer_name`
- `phone`
- `email`
- `delivery_type`
- `address` для доставки курьером
- `comment`

Дополнительные элементы:

- сводка по заказу
- итоговая сумма
- кнопка подтверждения заказа

Поведение:

- checkout реализован в один шаг
- при выборе `pickup` поле адреса скрывается и не валидируется
- при выборе `courier` поле адреса обязательно

### 5.5 Экран подтверждения заказа

Обязательные элементы:

- номер заказа
- список заказанных товаров
- итоговая сумма
- контактные данные покупателя
- выбранный способ получения
- ссылка "Вернуться в каталог"

## 6. Функциональные требования к админ-панели

### 6.1 Экран входа

Обязательные элементы:

- поля `login` и `password`
- кнопка входа
- сообщение об ошибке при невалидных данных

### 6.2 Экран списка товаров

Обязательные элементы:

- таблица товаров
- поиск по названию и SKU
- фильтр по категории и статусу активности
- колонка с остатком
- кнопка создания товара
- быстрые действия: редактировать, изменить остаток, деактивировать

### 6.3 Экран создания/редактирования товара

Поля формы:

- `sku`
- `name`
- `slug`
- `description`
- `category_id`
- `price`
- `stock_qty`
- `base_type`
- `wattage`
- `color_temperature_k`
- `luminous_flux_lm`
- `is_active`
- 'массив изображений товара'

### 6.4 Экран категорий

Обязательные элементы:

- список категорий
- создание категории
- редактирование названия и slug
- удаление категории, если к ней не привязаны товары

### 6.5 Экран списка заказов

Обязательные элементы:

- список заказов с номером, датой, суммой и статусом
- фильтры по статусу и дате
- поиск по номеру заказа, телефону или email
- переход в карточку заказа

### 6.6 Экран карточки заказа

Обязательные элементы:

- номер и дата заказа
- данные покупателя
- адрес и способ получения
- список позиций
- текущий статус
- история смены статусов
- селектор изменения статуса

## 7. User flow

### 7.1 Витрина

`Каталог -> Карточка товара -> Корзина -> Оформление заказа -> Подтверждение заказа`

Допустимый альтернативный путь:

`Каталог -> Корзина -> Оформление заказа -> Подтверждение заказа`

### 7.2 Админка

`Вход -> Список товаров -> Создание/редактирование товара`

`Вход -> Список заказов -> Карточка заказа -> Изменение статуса`

`Вход -> Категории -> Создание/редактирование категории`

## 8. Структура экранов в Figma

Прототипы должны быть собраны в одном Figma-файле на двух страницах:

1. `Storefront`
   Содержит экраны: каталог, карточка товара, корзина, checkout, confirmation.
2. `Admin`
   Содержит экраны: login, товары, форма товара, категории, заказы, карточка заказа.

Требования к прототипам:

- desktop-first, ширина рабочих фреймов 1440 px
- средняя детализация: реальные поля, таблицы, карточки, CTA-кнопки и подписи
- все переходы между ключевыми экранами обозначены стрелками или подписями
- у корзины и таблиц заказов должен быть виден реалистичный набор данных
- в админке должны быть отдельные состояния для списка и формы

## 10. API-границы микросервисов

### 10.1 `product-service`

Публичные endpoint:

- `GET /categories`
- `GET /products`
- `GET /products/{id}`

Админские endpoint:

- `POST /products`
- `PATCH /products/{id}`
- `DELETE /products/{id}`
- `PATCH /products/{id}/stock`
- `POST /categories`
- `PATCH /categories/{id}`
- `DELETE /categories/{id}`

### 10.2 `order-service`

Endpoint корзины:

- `GET /cart/{sessionId}`
- `POST /cart/{sessionId}/items`
- `PATCH /cart/{sessionId}/items/{itemId}`
- `DELETE /cart/{sessionId}/items/{itemId}`
- `DELETE /cart/{sessionId}`

Endpoint заказов:

- `POST /orders`
- `GET /orders/{orderId}`
- `GET /admin/orders`
- `GET /admin/orders/{orderId}`
- `PATCH /admin/orders/{orderId}/status`

### 10.3 `admin-panel-service`

Auth endpoint:

- `POST /auth/login`
- `GET /auth/me`
- `POST /auth/logout`

BFF endpoint админки:

- `GET /dashboard/summary`
- `GET /catalog/products`
- `POST /catalog/products`
- `PATCH /catalog/products/{id}`
- `DELETE /catalog/products/{id}`
- `GET /catalog/categories`
- `POST /catalog/categories`
- `PATCH /catalog/categories/{id}`
- `DELETE /catalog/categories/{id}`
- `GET /orders`
- `GET /orders/{id}`
- `PATCH /orders/{id}/status`

## 11. Нефункциональные требования

- время ответа для чтения каталога: до 500 мс при базе до 5000 товаров
- время ответа admin list endpoint: до 700 мс при базе до 10000 заказов
- API должны возвращать предсказуемые коды ошибок и JSON с полями `code`, `message`, `details`
- все формы должны иметь клиентскую и серверную валидацию
- пароль администратора хранится только в виде `password_hash`
- все изменения статуса заказа логируются в `order_status_history`
- все изображения товара передаются как URL или путь к CDN-ресурсу
- витрина должна корректно отображаться на десктопе и планшете; мобильная адаптация описывается правилами, но не прототипируется отдельными экранами

## 12. Acceptance criteria

### 12.1 Каталог

- пользователь видит список товаров, фильтры и сортировку
- фильтрация по категории, цоколю, мощности, цветовой температуре и наличию работает совместно
- товар без остатка помечен как недоступный

### 12.2 Карточка товара

- пользователь видит все обязательные характеристики товара
- нельзя добавить в корзину больше единиц, чем есть в остатке

### 12.3 Корзина

- пользователь может изменить количество и удалить товар
- итоговая сумма пересчитывается без перезагрузки
- пустая корзина имеет отдельное состояние и не позволяет перейти к заказу

### 12.4 Оформление заказа

- заказ оформляется без регистрации
- при `pickup` адрес не обязателен
- при `courier` адрес обязателен
- после успешного создания заказа показывается уникальный номер

### 12.5 Админка

- неавторизованный пользователь не может открыть административные endpoint
- администратор может создать, обновить и деактивировать товар
- администратор может обновить остаток товара
- администратор может просмотреть список заказов и изменить статус заказа

## 13. Негативные сценарии

- `GET /products/{id}` возвращает `404`, если товар не найден
- `POST /orders` возвращает `409`, если остаток хотя бы по одной позиции недостаточен
- `POST /orders` возвращает `400`, если корзина пуста или не переданы обязательные поля
- `POST /auth/login` возвращает `401`, если логин или пароль неверные
- `DELETE /categories/{id}` возвращает `409`, если категория используется хотя бы одним товаром
- `PATCH /admin/orders/{orderId}/status` возвращает `400`, если переход между статусами недопустим
